# CRIS-PRO 持续改进与功能开发系统

## 概述

CRIS-PRO 是 CRIS 的升级版本，在原有质量审查基础上，新增**新功能开发**能力，形成完整的闭环流程。

## 双循环架构

```
┌─────────────────────────────────────────────────────────────┐
│                    CRIS-PRO 双循环系统                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌──────────────────┐         ┌──────────────────┐        │
│   │   质量审查循环    │◄──────►│   功能开发循环    │        │
│   │   (CRIS-Q)       │  触发   │   (CRIS-F)       │        │
│   └──────────────────┘         └──────────────────┘        │
│          ▲                             ▲                    │
│          │                             │                    │
│          ▼                             ▼                    │
│   检查→审查→改进→总结           规划→分析→设计→实现          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

## 循环一：CRIS-Q 质量审查循环

### 4阶段

| 阶段 | 名称 | 做什么 |
|------|------|--------|
| **C** | Check（检查） | 全面检查代码、配置、依赖、文档等 |
| **R** | Review（审查） | 深入分析发现的问题，评估影响 |
| **I** | Improve（改进） | 实施修复、优化、重构 |
| **S** | Summary（总结） | 记录完成的工作，更新进度文档 |

### 执行方式

- **批量执行**：每10轮为一批
- **全面覆盖**：前端、后端、基础设施
- **发现问题 → 立即修复**

---

## 循环二：CRIS-F 功能开发循环

### 4阶段

| 阶段 | 名称 | 做什么 |
|------|------|--------|
| **P** | Plan（规划） | 规划新功能、新页面、新模块 |
| **A** | Analyze（分析） | 需求分析、优先级排序、影响评估 |
| **D** | Design（设计） | 架构设计、API设计、组件设计 |
| **B** | Build（构建） | 编码实现、测试、文档 |

### 执行方式

- **由需求触发**：用户提出、功能审查中发现、规划中发现
- **完整流程**：每个功能走完 P-A-D-B 全流程
- **与 CRIS-Q 协同**：新功能开发后进入质量审查

## CRIS-PRO 执行规则

### 1. 日常执行

```
每日 / 定期执行：
├── CRIS-Q 循环（10轮质量检查）
│   ├── 检查代码质量
│   ├── 检查构建状态
│   ├── 检查文档完整性
│   └── 发现问题 → 立即修复
│
└── 如果有新需求 → 触发 CRIS-F
    ├── P: 规划新功能
    ├── A: 分析需求
    ├── D: 设计方案
    └── B: 实现功能
```

### 2. GitHub 同步规则

| 操作 | 时机 | 命令 |
|------|------|------|
| 代码变更 | 随时 | `git add` + `git commit` + `git push` |
| PROGRESS.md | 每10轮更新 | `git add` + `git commit` + `git push` |
| 功能文档 | 每个新功能完成 | `git add` + `git commit` + `git push` |
| 架构文档 | 重大设计变更 | `git add` + `git commit` + `git push` |

### 3. 文档更新规则

| 文档 | 更新时机 | 内容 |
|------|----------|------|
| PROGRESS.md | 每10轮CRIS-Q | 记录每轮检查了什么 |
| CHANGELOG.md | 每个新功能 | 记录功能变更 |
| ARCHITECTURE.md | 架构变更时 | 更新系统架构 |
| README.md | 需要时 | 更新项目说明 |

### 4. 新功能开发流程

```
用户提出需求
    │
    ▼
┌─────────────┐
│  P - 规划   │  → 功能描述、目标、范围
└─────────────┘
    │
    ▼
┌─────────────┐
│  A - 分析   │  → 优先级、依赖、风险评估
└─────────────┘
    │
    ▼
┌─────────────┐
│  D - 设计   │  → API设计、组件设计、数据库设计
└─────────────┘
    │
    ▼
┌─────────────┐
│  B - 构建   │  → 编码、测试、文档
└─────────────┘
    │
    ▼
  进入 CRIS-Q 质量审查
    │
    ▼
  推送 GitHub
```
