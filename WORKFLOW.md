# Vox (内容生成Agent) - 自主工作流

> ⚠️ **这是作品集项目，不是demo**
> ⚠️ 每个功能都要有完整的前端界面，能实际演示
> ⚠️ 前后端同等重要，缺一不可

---

## 核心原则：作品集标准

> 核心理念：像 Sigma 学习，自主循环、自主进化、持续改进
> **但更重要的是：做出能拿出手给面试官看的作品**

**作品标准**：
1. 后端Agent能正常工作（调用API有返回）
2. 前端有完整的用户界面（不是空白页面）
3. 前后端能联调（不是各自独立）
4. 能拿出手给面试官看（不是半成品）

**检查清单 - 每实现一个功能必须同时完成**：
- [ ] 后端：Agent代码 + API接口
- [ ] 前端：操作界面 + 状态展示
- [ ] 联调：前后端能正常通信
- [ ] 体验：加载状态、错误处理
- [ ] 演示：功能可以实际运行展示

**必须思考的问题**（不只是完成功能）：
1. **Vox 的核心竞争力是什么？** 比 MoneyPrinterTurbo 好在哪儿？
2. **数据从哪来？** 产品信息、平台数据、文案内容...捏！
3. **为什么要做这个功能？** 是因为参考项目有这个，还是真的有需求？
4. **竞品会怎么评价？** 面试官问"和 xxx 有什么区别"怎么答？

**常见错误**：
- ❌ 只写后端，不管前端
- ❌ 前端只有一个空白页面
- ❌ 前后端不通信，各玩各的
- ❌ 功能实现了但界面无法演示
- ❌ 觉得"能跑通就行" - **作品级标准是"能拿出手"**
- ❌ 觉得"没什么可优化的" - **去用参考项目，找差距**

**数据捏造**：
- 可以用 mock 数据，只要真实感强
- 产品信息、多平台文案、审核结果都可以捏
- 重点是呈现产品思路，不是真的接 API

## Vox 产品特性

> 结合总指导文件的共性原则与 Vox 产品自身特性

### Vox 核心竞争力

| 维度 | 内容 |
|------|------|
| **核心竞争** | 一键生成多平台营销内容（小红书/抖音/公众号/朋友圈） |
| **vs 参考项目** | 比 MoneyPrinterTurbo 更完整的文案→审核→导出工作流 |
| **杀手锏** | 文案生成→审核→导出完整流程，违规词检测，市场调研集成 |

### Vox 价值主张

1. **多平台一致性**：一份产品信息，生成适配4个平台风格的文案
2. **质量保障**：内置违规词检测 + 逻辑审核，不是只生成就完事
3. **市场洞察**：集成 Tavily 调研，生成的内容有数据支撑
4. **导出灵活**：支持多格式导出（HTML/Markdown/JSON），适配不同发布渠道

### Vox 必须思考的问题

在做每个功能之前，必须能回答：

1. **这个功能强化了 Vox 哪个竞争力？**
   - 是多平台适配？质量审核？市场调研？还是导出能力？
   - 如果都不沾边，为什么要做？

2. **竞品（MoneyPrinterTurbo等）有没有这个功能？**
   - 有：Vox 做得更好吗？好在哪儿？
   - 没有：为什么没有？是因为不需要还是因为难？

3. **这个功能的数据从哪来？**
   - 产品信息 → 用户输入还是捏造？
   - 平台数据 → 真实API还是模拟？
   - 文案内容 → AI生成还是模板？

4. **面试官会问什么？**
   - "和 MoneyPrinterTurbo 有什么区别？"
   - "违规词检测怎么做的？"
   - "多平台文案怎么保证质量？"
   - "市场调研集成怎么用？"

### Vox 前端界面标准

- 每个页面必须有完整功能（不是空白占位）
- 前后端必须联调（不能各玩各的）
- 加载状态、错误处理、toast 提示必须完善
- 颜色主题统一：活力橙 `#FF6B35` 为主色

### Vox 后端模块规范

```
backend/
├── agents/
│   ├── planner.py      # 内容规划（平台选择、风格定位）
│   ├── copywriter.py   # 文案生成（多平台适配）
│   ├── reviewer.py     # 质量审核（违规词、逻辑）
│   └── exporter.py     # 导出模块（多格式）
├── services/
│   ├── market_research.py  # Tavily 调研集成
│   └── keyword_check.py    # 违规词检测
└── api/
    └── routes/         # FastAPI 路由
```

### 循环迭代机制

> 每次循环结束后，检查是否还有可改进匹配项

```
                        ↓ 循环结束
                        │
        ┌───────────────┴───────────────┐
        ↓                               ↓
   ┌─────────────────────────────────┐   ┌────────────────────┐
   │ 检查：还有可改进的匹配项吗？      │   │ 循环已达 10次？     │
   │ - 功能架构图中的模块还没实现完？   │   │ 暂停，等待用户新任务 │
   │ - 参考项目中还有可借鉴的设计？     │   └────────────────────┘

   │ - 测试还有不通过的地方？           │         ↓
   └─────────────────────────────────┘   ┌────────────────────┐
        ↓                               │ 暂停，等待用户新任务 │
   ┌────────────┐                       └────────────────────┘
   │ 有，       │
   │ 说明哪些   │
   │ 还没做好   │
   └────────────┘
        ↓
   ┌─────────────────────────────────────────┐
   │ 继续迭代：回到对应步骤继续改              │
```

## 九、自主循环改进机制 (CRIS - Continuous Review & Improvement System)

> **核心理念**：像产品团队一样运营，每个功能都要问"真有用吗？真有必要吗？能更好吗？"

### 9.1 循环节奏

```
┌─────────────────────────────────────────────────────────────┐
│  自主循环 (CRIS Loop)                                      │
│  每轮执行以下四个阶段，形成闭环                             │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  Phase 1: 🔍 巡检 (Review) - 5分钟                        │
│  - 扫一遍刚实现的功能，能跑通吗？                           │
│  - 查 API 端点响应正常吗？                                 │
│  - 看前端组件加载正常吗？                                   │
│  - 问：这个功能真有必要吗？                                 │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  Phase 2: 🐛 找 Bug (Bug Hunt) - 10分钟                    │
│  - 跑一遍功能流程，找异常                                   │
│  - 检查错误处理、边界情况                                   │
│  - 检查组件卸载保护、内存泄漏                               │
│  - 问：有哪些没想到的坑？                                   │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  Phase 3: ✨ 优化 (Optimize) - 10分钟                      │
│  - 代码有没有重复可以抽象？                                 │
│  - 有没有更简洁的实现？                                     │
│  - 前端交互可以更流畅吗？                                   │
│  - 问：能做得更好的地方？                                   │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  Phase 4: 🚀 新功能 (New Feature) - 15分钟                  │
│  - 基于巡检发现的需求                                       │
│  - 基于竞品分析的机会                                       │
│  - 用户体验提升的小功能                                       │
│  - 问：下一个最有价值的东西是什么？                         │
└─────────────────────────────────────────────────────────────┘
                        ↓
                        ↓ 循环
```

### 9.2 巡检清单 (Review Checklist)

每次巡检必须检查：

**后端检查**：
- [ ] API 端点能正常调用吗？
- [ ] 返回数据格式对吗？
- [ ] 错误处理有返回友好信息吗？
- [ ] 日志记录完整吗？

**前端检查**：
- [ ] 组件能正常渲染吗？
- [ ] 加载状态有显示吗？
- [ ] 错误状态有处理吗？
- [ ] Toast 提示有吗？
- [ ] 组件卸载保护做了吗？（isMountedRef）

**联调检查**：
- [ ] 前后端能正常通信吗？
- [ ] 请求参数对吗？
- [ ] 响应数据正确解析了吗？

**产品价值检查**：
- [ ] 这个功能真有人用吗？
- [ ] 不做会怎样？
- [ ] 竞品有类似功能吗？
- [ ] 比竞品好在哪里？

### 9.3 Bug Hunt 重点

```
重点检查区域：
1. 异步操作 - isMountedRef 保护做了吗？
2. API 调用 - try/catch 全覆盖了吗？
3. 前端状态 - setState 在 unmount 后调用了吗？
4. 边界情况 - 空数据、极端值处理了吗？
5. 类型安全 - TypeScript 类型都对吗？
```

### 9.4 新功能优先级

```
优先级判断框架：

P0 - 必须做（核心流程）
  └─ 影响用户无法完成主要任务

P1 - 应该做（体验优化）
  └─ 明显提升用户体验

P2 - 可以做（锦上添花）
  └─ 有余力时做

P3 - 考虑砍掉（价值不足）
  └─ 做不做影响不大
```

### 9.5 循环执行命令

```bash
# 开始一轮 CRIS 循环
# 1. 巡检 - 检查最近实现的功能
# 2. Bug Hunt - 找问题
# 3. 优化 - 改进代码
# 4. 新功能 - 实现一个小改进

# 查看当前项目状态
git status

# 查看最近完成的功能
git log --oneline -10

# 运行后端测试
cd backend && python -m pytest -v --tb=short 2>&1 | head -50

# 运行前端构建
cd frontend && npm run build 2>&1 | tail -30
```

每轮循环结束，更新 PROGRESS.md：


### 9.7 功能审查触发条件

遇到以下情况，触发深度审查：

1. **功能长期不用** - 超过 5 轮循环都没涉及
2. **Bug 反复出现** - 同一模块 Bug 超过 3 次
3. **复杂度膨胀** - 模块代码超过 500 行
4. **用户体验差** - 加载超过 3 秒，无反馈
5. **价值不明确** - 无法回答"为什么做这个"

深度审查后决定：
- 保留：继续维护
- 优化：重构改进
- 合并：与其他功能合并
- 移除：不再维护
