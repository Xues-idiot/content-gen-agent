# Vox (内容生成Agent) - 自主工作流

## 自主循环机制

> 核心理念：像 Sigma 学习，自主循环、自主进化、持续改进

---

## 一、每次接收到任务的循环流程

```
┌─────────────────────────────────────────────────────────────┐
│  接收任务                                                    │
│  输入：产品信息 + 目标平台                                   │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤0: 检查功能框架                                         │
│  - 读 ../_FEATURE-FRAMEWORK.md                              │
│  - ⚠️ 只读 "Vox 任务清单" 部分，不要读其他项目的功能       │
│  - ⚠️ 只添加 Vox 自己的功能，不要借鉴其他项目                │
│  - 定期自主循环时检查是否有新的GitHub发现需要添加           │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤1: 理解任务                                             │
│  - 读 ARCHITECTURE.md 理解当前架构                        │
│  - 读 INSTRUCTIONS.md 理解产品定位                         │
│  - 问自己：这个产品适合什么内容风格？                        │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤2: 规划实现                                            │
│  - 需要改哪些模块？                                          │
│  - 需要新增哪些函数？                                        │
│  - 需要参考 Sigma 哪些 Skills/Agents？                      │
│  - 更新 ARCHITECTURE.md（新增模块）                         │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤3: GitHub 学习（如果需要）                            │
│  - 去 _reference/ 研究参考项目                             │
│  - 搜索 GitHub 相关项目找灵感                               │
│  - 如果有新的好的参考，clone 到 _reference/                │
│  - 记录学到的东西到 _LEARNING.md                           │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤4: 实现代码                                            │
│  - 按 ARCHITECTURE.md 的模块实现                           │
│  - 遵循 Sigma 的 Skills 规范                              │
│  - 遵循 ARCHITECTURE.md 的格式写注释                       │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤5: 测试验证                                            │
│  - 运行测试用例                                             │
│  - 文案质量如何？违规词检测通过？                            │
│  - 配图建议是否合适？                                        │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤6: 更新文档                                            │
│  - 更新 ARCHITECTURE.md（如果新增模块）                    │
│  - 更新 _LEARNING.md（记录学到的）                         │
│  - 更新 PROGRESS.md（记录本轮做了什么）                    │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤7: 自我审计                                            │
│  - 代码质量：遵循规范？模块化？                              │
│  - 功能完整：所有功能都实现了？                              │
│  - 文档完整：注释、ARCHITECTURE、SKILL都更新了？            │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤8: 版本管理                                            │
│  - git add . && git commit -m "描述本次完成的工作"        │
│  - git push 推送到 GitHub 仓库                            │
│  - 确保所有重要更改都已提交和推送                          │
└─────────────────────────────────────────────────────────────┘
                        ↓
                        ↓ 循环结束
```

## 二、学习实现分支（给定功能+参考项目+功能架构图）

> 触发条件：用户明确要求"学习实现"模式，并提供：
> - 要实现的功能描述
> - 参考项目路径 (`_reference/` 或 GitHub)
> - 功能架构图 (`../_FEATURE-FRAMEWORK.md` 中对应部分)

**与旧分支的区别**：
- 旧分支：用户提供任务描述 → Agent自主分析、自主发现新功能、自主规划
- 新分支：用户给定具体功能 + 参考项目 + 功能架构图 → Agent学习参考 → 按架构图实现

```
┌─────────────────────────────────────────────────────────────┐
│  接收学习实现任务                                           │
│  输入：功能描述 + 参考项目 + 功能架构图                       │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤0: 理解功能需求                                       │
│  - 读用户给的功能描述，明确要实现什么                        │
│  - 读 ../_FEATURE-FRAMEWORK.md 中该功能对应的架构设计     │
│  - 读 ARCHITECTURE.md 了解现有架构                         │
│  - ⚠️ 不要自主添加功能，只实现用户给定的范围                │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤1: 研究参考项目                                        │
│  - 去 _reference/ 深入研究用户指定的参考项目               │
│  - 分析参考项目的核心实现逻辑                               │
│  - 提取可复用的设计模式和代码结构                           │
│  - 记录学到的东西到 _LEARNING.md                           │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤2: 对照功能架构图                                      │
│  - 细读 ../_FEATURE-FRAMEWORK.md 中该功能的架构设计       │
│  - 确认参考项目的设计与架构图是否匹配                       │
│  - 如有不符，以功能架构图为准                               │
│  - 更新 ARCHITECTURE.md（如需新增模块）                     │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤3: 实现代码                                            │
│  - 按功能架构图的模块划分实现代码                           │
│  - 参考 _reference/ 中的项目结构和写法                      │
│  - 遵循 Sigma 的 Skills 规范                              │
│  - 遵循 ARCHITECTURE.md 的格式写注释                       │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤4: 测试验证                                            │
│  - 运行 npm run build / python -m pytest                   │
│  - 功能是否按功能架构图正常运行？                           │
│  - API 端点与后端路由一致？                                │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤5: 更新文档                                            │
│  - 更新 ARCHITECTURE.md（如果新增模块）                    │
│  - 更新 _LEARNING.md（记录学到了什么）                     │
│  - 更新 PROGRESS.md（记录本轮做了什么）                     │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤6: 自我审计                                            │
│  - 代码质量：遵循规范？模块化？                             │
│  - 功能完整：功能架构图中的模块都实现了？                   │
│  - 文档完整：注释、ARCHITECTURE、SKILL都更新了？           │
│  - 冲突检查：API路由一致？依赖正确？命名无冲突？            │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤7: 版本管理                                            │
│  - git add . && git commit -m "feat(Vox): 实现 [功能名]" │
│  - git push 推送到 GitHub 仓库                            │
│  - 确保所有重要更改都已提交和推送                           │
└─────────────────────────────────────────────────────────────┘
                        ↓
                        ↓ 循环结束
```

**Skills 学习要点**（同步进行）：
- 读 pm-agent-forge/skills/ 中相关的 Skills
- 把相关 Skills 的检查清单应用到实现中
- 如发现 Skills 有缺失，记录到 _LEARNING.md 供 Sigma 扩展

### 循环迭代机制

> 每次循环结束后，检查是否还有可改进匹配项

```
                        ↓ 循环结束
                        │
        ┌───────────────┴───────────────┐
        ↓                               ↓
   ┌─────────────────────────────────┐   ┌────────────────────┐
   │ 检查：还有可改进的匹配项吗？      │   │ 循环已达 10次？     │
   │ - 功能架构图中的模块还没实现完？   │   │ 暂停，等待用户新任务 │
   │ - 参考项目中还有可借鉴的设计？     │   └────────────────────┘
   │ - 代码还有优化空间？               │
   │ - 测试还有不通过的地方？           │         ↓
   └─────────────────────────────────┘   ┌────────────────────┐
        ↓                               │ 暂停，等待用户新任务 │
   ┌────────────┐                       └────────────────────┘
   │ 有，       │
   │ 说明哪些   │
   │ 还没做好   │
   └────────────┘
        ↓
   ┌─────────────────────────────────────────┐
   │ 继续迭代：回到对应步骤继续改              │
   │ - 功能模块没实现完 → 步骤3（实现代码）   │
   │ - 设计有更好方案   → 步骤1（研究参考）   │
   │ - 代码需优化       → 步骤3（实现代码）   │
   │ - 测试不通过       → 步骤4（测试验证）   │
   └─────────────────────────────────────────┘
        ↓
    回到"步骤0: 理解功能需求"继续循环
```

## 三、定期自主循环（无新任务时）

当没有新任务时，每轮自动执行：

```
┌─────────────────────────────────────────────────────────────┐
│  定期循环 (每轮自动执行)                                    │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤0: 再检查功能框架                                      │
│  - ⚠️ 只确认 "Vox 任务清单" 是否有更新                      │
│  - ⚠️ 不要看其他项目的功能清单                              │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  1. GitHub 趋势扫描                                        │
│  - 搜索 AI content generation / marketing copy 相关项目    │
│  - 找有没有更好的内容生成相关工具                            │
│  - 有就 clone 到 _reference/                              │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  2. Sigma Skills 学习                                      │
│  - 读 pm-agent-forge/skills/ 相关 Skills                   │
│  - 看有没有可以应用到内容生成的                             │
│  - 有就更新自己的实现                                       │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  3. 代码优化                                                │
│  - 检查有没有重复代码                                       │
│  - 检查有没有可以抽象的模块                                  │
│  - 检查有没有更优雅的实现方式                                │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  4. 功能增强                                                │
│  - 看参考项目有没有新功能                                    │
│  - 实现一个小的功能增强                                      │
│  - 记录到 PROGRESS.md                                      │
└─────────────────────────────────────────────────────────────┘
```

## 七、循环终止条件

可以终止的情况：
1. 用户说"满意了"或"可以了"
2. 循环 10 次后自动暂停，等待用户新任务
3. 遇到无法解决的技术难题，记录并暂停
